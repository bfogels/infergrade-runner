# InferGrade v0.1 Architecture

## Purpose

This document turns the current InferGrade strategy into a concrete architecture target for v0.1.

The key architectural decision is that InferGrade is not one deployable thing.

It is:

- an open-source `Runner`
- a hosted `Hub`
- plus shared schemas and bundle contracts between them

## v0.1 User Promise

InferGrade v0.1 should let a user do both of these cleanly:

1. sign into the hosted hub, connect Hugging Face, generate a run, execute it locally or in cloud, and see the result appear automatically
2. install the open-source runner directly, execute a benchmark without adopting the hosted hub first, and still produce a standards-compliant bundle

## Architecture Overview

## 1. InferGrade Runner

The runner is the execution plane.

### Responsibilities

- consume run configs
- resolve artifacts and cache them locally
- launch backend and benchmark containers
- collect deployment telemetry and capability scores
- emit normalized result bundles
- upload results to the hub when configured
- support preflight, progress tracking, and resume

### Distribution

- open-source
- container-first
- runnable on local machines and cloud workers

## 2. InferGrade Hub

The hub is the hosted control plane.

### Responsibilities

- single sign-on
- contributor identity and accounts
- connected Hugging Face credentials
- recommendation and compare surfaces
- run-config generation
- run/job management
- result hosting and community browsing
- trust policy and moderation

### Distribution

- canonical hosted product
- not optimized for broad self-hosting in v0.1

## 3. Shared Contract Layer

The runner and hub should communicate through versioned artifacts:

- run config schema
- result bundle schema
- ontology schema
- validation rules

This is the stable seam in the system.

## v0.1 Core Flows

## Hosted Local Flow

1. User signs into the hub.
2. User connects Hugging Face.
3. Hub resolves model/artifact options and generates a run config.
4. User launches the runner locally.
5. Runner executes the run and uploads the bundle.
6. Hub stores, validates, and surfaces the result.

## Hosted Cloud Flow

1. User signs into the hub.
2. User chooses `Run in cloud`.
3. Hub creates a run job and launch plan.
4. A runner instance executes on a cloud worker.
5. Bundle uploads automatically.
6. Hub shows progress and publishes the result.

## Standalone Runner Flow

1. User installs the runner.
2. User runs a config or request locally.
3. Runner emits a local bundle.
4. User optionally uploads later to the hub.

## Identity and Credential Model

v0.1 should clearly separate identity from execution.

## Hub-Owned Identity

The hub should own:

- SSO
- contributor profiles
- connected Hugging Face credentials
- later cloud provider connections

## Runner-Owned Execution State

The runner should own:

- local environment capture
- artifact cache
- bundle state
- progress and resume state

The runner should use scoped credentials from the hub where needed, but it should not become the long-lived identity system.

## Hugging Face Integration

This should be a first-class v0.1 hub capability.

The hub should use Hugging Face metadata and connected credentials to:

- resolve candidate artifacts
- infer ontology hints
- support gated/private artifacts later
- reduce manual form entry dramatically

The runner should then execute the resolved plan.

## v0.1 Runner Scope

The runner should support:

- `llama.cpp` first
- `vLLM` next
- local container execution
- cloud worker execution through the same run-job contract
- artifact resolution for `hf://`, `http(s)://`, `file://`, and local paths
- bundle upload

## v0.1 Hub Scope

The hub should support:

- hosted authentication and basic account identity
- run-config generation
- result ingestion and validation
- trust badges and comparison classes
- recommendation and compare flows
- `My Runs` and contributor activity
- cloud run planning
- agent-friendly read and decision APIs

## Canonical Analysis Slices

The hub should still publish stricter comparison slices.

The first official slice remains a strong candidate:

- single NVIDIA GPU
- 24 GB VRAM class
- `general_assistant` and `agentic_coding`
- `llama.cpp` first, then `vLLM`

But this is a hosted analysis concern, not a limit on what the runner may execute.

## Data Model

The v0.1 data model remains joined and ontology-aware:

- configuration identity
- artifact identity
- runtime binding
- hardware telemetry
- capability results
- deployment telemetry
- cost
- verification
- derived recommendations

The hosted hub consumes this contract.
The runner emits it.

## Non-Goals for v0.1

- broad hub self-hosting
- making the hub the only way to use InferGrade
- full provider-managed cloud orchestration across many providers
- trying to support every backend immediately
- replacing the open runner with a pure SaaS-only architecture

## Practical Repo Interpretation

In the current monorepo:

- [python/runner-core](../python/runner-core) is the open runner
- [services/api](../services/api) and [apps/web](../apps/web) are hub scaffolding
- [schemas](../schemas) is the shared contract layer

That means the repo can continue to evolve in one place while the product boundary stays clear.

## What v0.1 Must Prove

v0.1 is successful if it proves:

1. the runner is useful and trustworthy on its own
2. the hosted hub meaningfully reduces friction through identity, Hugging Face integration, and community evidence
3. the runner-hub contract is stable enough that runs feel simple rather than bespoke
